# Дополнительное техническое задание

Дата фиксации: 24 июля 2026 года.

Документ дополняет исходное техническое задание FastCompute.NET и определяет
требования для следующих этапов разработки. При противоречии с текущими
эвристиками выбора backend приоритет имеют требования этого документа.

## 1. Пользовательские методы

### 1.1. CPU backend

Для явно выбранных `Scalar` и `ParallelCpu` должна быть доступна обработка
массива произвольным пользовательским delegate без преобразования метода во
внутренний compute IR.

Пример целевого API:

```csharp
static float Transform(float value)
{
    return MathF.Sin(value) + CustomCalculation(value);
}

Compute.RunDelegate(
    source,
    Transform,
    ComputeBackendKind.ParallelCpu);
```

Окончательное имя API определяется при проектировании. Следует использовать
отдельное имя метода, чтобы избежать неоднозначности между `Func<T, TResult>`
и `Expression<Func<T, TResult>>`.

Требования:

- `Scalar` вызывает delegate последовательно;
- `ParallelCpu` вызывает delegate параллельно для независимых диапазонов;
- пользователь отвечает за потокобезопасность delegate в `ParallelCpu`;
- исключения пользовательского метода не подавляются;
- cancellation проверяется библиотекой между элементами или блоками;
- API не обещает SIMD-векторизацию пользовательского метода;
- `Auto` для произвольного delegate выбирает только Scalar или Parallel CPU.

### 1.2. SIMD backend

Произвольный scalar delegate нельзя считать SIMD-совместимым. Для SIMD
разрешены только операции, которые могут быть преобразованы в поддерживаемый
vector IR.

При явно выбранном `Simd` и неподдерживаемом методе библиотека должна:

- выбросить понятное исключение; либо
- использовать явно запрошенную пользователем политику fallback.

Молчаливое выполнение scalar delegate под именем SIMD запрещено.

### 1.3. GPU backend

GPU не обязан поддерживать произвольные пользовательские методы. На GPU
запрещены или технически невозможны managed-аллокации, I/O, сервисы, delegates,
произвольные ссылочные объекты и другие конструкции, несовместимые с ILGPU.

В перспективе допускается отдельный механизм регистрации GPU-совместимых
методов через source generator, атрибут или заранее скомпилированный kernel.
Такой механизм должен проверять допустимое подмножество C# во время сборки.

Явное указание `Backend = Gpu` само по себе не делает произвольный метод
GPU-совместимым.

### 1.4. Явный extension API

Для сценариев с полным ручным управлением доступен LINQ-подобный extension:

```csharp
float[] result = source.RunExplicit(
    value => GpuMath.Sin(value),
    new ComputeOptions
    {
        Backend = ComputeBackendKind.Gpu,
        GpuContext = context
    });
```

Требования:

- `Backend.Auto` запрещён;
- Scalar, Parallel CPU, SIMD или GPU выполняются ровно по явному запросу;
- `AllowFallback` не заменяет явно выбранный backend;
- точный GPU задаётся через `GpuContext`;
- SIMD и GPU по-прежнему принимают только поддерживаемый compute IR;
- произвольные пользовательские методы выполняются через отдельный
  `RunDelegate` только на CPU.

Название `RunExplicit` используется вместо `RunUnsafe`, поскольку API не
нарушает безопасность памяти, а требует явного выбора backend.

## 2. In-place обработка

### 2.1. Unary Map

Необходимо добавить явный API, записывающий результат в исходный массив:

```csharp
Compute.RunInPlace(
    source,
    value => value + 2.0f,
    options);
```

Использование отдельного метода обязательно. Обычный `Compute.Run` сохраняет
текущий контракт и возвращает новый массив.

In-place Map должен поддерживаться:

- Scalar CPU;
- Parallel CPU;
- SIMD;
- GPU one-shot;
- GPU chunked execution.

Операция разрешена только для выражений, где каждый выходной элемент зависит
от соответствующего входного элемента и констант.

### 2.2. Zip

Для Zip необходимо явно указывать перезаписываемый массив:

```csharp
Compute.ZipInPlace(
    target: left,
    right,
    (x, y) => x + y,
    options);
```

Длины массивов должны совпадать. Неявное обрезание запрещено.

### 2.3. Поведение при ошибках

In-place операция может изменить часть массива до:

- отмены через `CancellationToken`;
- исключения пользовательского метода;
- ошибки backend;
- ошибки передачи данных.

Это поведение должно быть явно задокументировано. Гарантия отката исходного
массива не предоставляется, если пользователь отдельно не запросил
транзакционный режим с дополнительным буфером.

### 2.4. GPU-resident граф

In-place изменение `ComputeBuffer<T>` допускается только при безопасном
владении allocation.

Если старое значение используется другой веткой execution graph, применяется:

- copy-on-write; либо
- отказ от in-place с понятной диагностикой.

Нельзя повреждать данные, необходимые другим узлам ленивого графа.

## 3. Порционная GPU-обработка

### 3.1. Общие требования

Если все входы, выход и временные данные не помещаются в доступную видеопамять,
операция должна иметь возможность выполняться chunks.

Поддерживаемые операции:

- Map;
- Zip;
- in-place Map;
- in-place Zip;
- Reduction;
- Histogram после его реализации.

Для независимых Map/Zip chunks обрабатываются в порядке индексов, а результат
записывается в соответствующий диапазон выходного массива.

### 3.2. Планирование памяти

Перед выбором GPU planner должен оценивать:

- доступную и общую память accelerator;
- размер всех входных GPU-буферов;
- размер результата;
- размер программы ядра;
- промежуточные buffers reduction или Histogram;
- резерв безопасности;
- allocations, уже принадлежащие `ComputeContext`.

Планировщик не должен выбирать non-chunked GPU-выполнение, если расчётный
рабочий набор не помещается в установленный memory budget.

Предлагаемые настройки:

```csharp
public sealed class ComputeOptions
{
    public bool EnableGpuChunking { get; init; } = true;
    public long? GpuMemoryBudgetBytes { get; init; }
    public int? GpuChunkElementCount { get; init; }
}
```

Точные имена и расположение настроек определяются при проектировании API.

Автоматический размер chunk должен оставлять безопасный запас видеопамяти.
Если allocation не прошёл из-за изменения доступной памяти, разрешается
уменьшить chunk и повторить allocation ограниченное число раз.

### 3.3. Streaming pipeline

После корректной базовой реализации допускается double или triple buffering:

```text
upload chunk N+1
execute chunk N
download chunk N-1
```

Передача и выполнение могут перекрываться только при наличии безопасной
поддержки нескольких ILGPU streams. Базовая последовательная chunked
реализация должна быть добавлена и протестирована первой.

### 3.4. Reduction

Каждый chunk формирует частичный результат на GPU. Затем частичные результаты
объединяются дополнительной reduction.

Необходимо сохранить документированную семантику:

- `Sum` пустого массива возвращает ноль;
- `Min`, `Max` и `Average` пустого массива выбрасывают исключение;
- `NaN` обрабатывается согласованно с существующими backend;
- допускается floating-point погрешность из-за изменения порядка сложения.

### 3.5. In-place chunking

Для unary Map предпочтителен следующий режим:

```text
CPU chunk
    -> upload в один device buffer
    -> in-place GPU kernel
    -> download в исходный CPU chunk
```

Это уменьшает потребление host и device memory. Обработанные диапазоны могут
быть изменены до возникновения последующей ошибки.

## 4. Исправление автоматического planner

Текущий выбор GPU только по размеру массива и категории выражения недостаточен.
Простое выражение над CPU-resident массивом может выполняться на GPU медленнее
из-за upload/download, даже для очень большого массива.

Planner должен учитывать:

- находится ли источник уже на GPU;
- объём upload и download;
- arithmetic intensity выражения;
- доступную видеопамять;
- необходимость chunking;
- измеренные характеристики CPU, SIMD, Parallel CPU и GPU;
- пользовательские пороги и явный backend.

Консервативное правило по умолчанию:

- простая арифметика над CPU-resident данными не направляется на дискретный GPU
  без подтверждённой модели производительности;
- для больших простых массивов сравниваются SIMD и Parallel CPU;
- тяжёлые выражения могут использовать GPU после проверки memory budget;
- GPU-resident данные остаются приоритетным сценарием GPU;
- явно выбранный backend не заменяется молча другим backend.

Если выбран `Auto` и GPU-план невозможен из-за памяти, библиотека:

1. пытается построить chunked GPU-план, если он разрешён и оправдан;
2. иначе выбирает доступный CPU backend;
3. записывает причину выбора в diagnostics.

Если пользователь явно выбрал `Gpu`, библиотека:

1. использует обычное GPU-выполнение, если оно помещается в memory budget;
2. иначе использует chunked GPU-выполнение, если оно разрешено;
3. иначе выбрасывает понятное исключение;
4. не переходит на CPU без отдельной явной политики пользователя.

## 5. Diagnostics

Диагностику необходимо расширить следующими полями или эквивалентными
структурами:

- причина выбора backend;
- причина отклонения GPU;
- оценённый device working set;
- применённый memory budget;
- количество chunks;
- размер chunk;
- количество переданных на GPU и с GPU bytes;
- признак in-place выполнения;
- признак fallback;
- исходный и фактически использованный backend.

Диагностика должна позволять объяснить, почему простое выражение было выполнено
на CPU, а тяжёлое выражение того же размера - на GPU.

## 6. Критерии приёмки

### Пользовательские методы

- произвольный delegate выполняется на Scalar CPU;
- потокобезопасный delegate выполняется на Parallel CPU;
- Auto для delegate не выбирает SIMD или GPU;
- неподдерживаемый принудительный SIMD/GPU выдаёт понятную ошибку;
- исключения delegate не подавляются.

### In-place

- результат совпадает с out-of-place версией;
- возвращаемый или изменяемый объект является исходным массивом;
- Scalar, Parallel, SIMD и GPU дают согласованный результат;
- проверены пустой массив, один элемент, tail SIMD и большие массивы;
- cancellation и исключение документированно допускают частичный результат;
- ветвление GPU graph не повреждает другие ветки.

### Chunked GPU

- массив больше доступного memory budget обрабатывается без device OOM;
- chunks не пропускаются и не пересекаются;
- Map/Zip совпадают со Scalar с допустимой погрешностью;
- chunked reduction совпадает с существующей GPU reduction;
- in-place chunking изменяет исходный массив корректно;
- allocation failure приводит к уменьшению chunk или понятной ошибке;
- Auto не выбирает GPU, когда передача заведомо дороже CPU выполнения.

### Performance

Добавить BenchmarkDotNet-сценарии:

- простая арифметика: Scalar, SIMD, Parallel CPU, GPU и Auto;
- тяжёлая арифметика: те же backend;
- out-of-place и in-place;
- один большой GPU allocation и chunked GPU;
- первый и повторный запуск;
- CPU-resident и GPU-resident данные.

Performance gate должен проверять, что Auto не медленнее лучшего допустимого
CPU-варианта более чем на согласованный допуск для простых CPU-resident
операций.

## 7. Этапы реализации

1. Исправить Auto planner: memory guard и запрет неоправданного simple GPU.
   Реализовано 24 июля 2026 года: simple GPU отключён по умолчанию, добавлена
   консервативная оценка working set, 75% safety budget, пользовательский
   `GpuMemoryBudgetBytes` и объяснение решения в diagnostics. Live free-memory
   accounting и chunking остаются последующими этапами.
2. Добавить CPU delegate API.
   Реализовано 24 июля 2026 года: добавлены `RunDelegate` и
   `RunDelegateWithDiagnostics`. Delegate исполняется напрямую на Scalar или
   Parallel CPU без преобразования в compute IR. Auto выбирает только между
   этими backend; явные SIMD и GPU отклоняются с
   `ComputeBackendNotSupportedException`.
3. Добавить явный LINQ-подобный extension API без Auto.
   Реализовано 24 июля 2026 года: `float[]` получил extensions `RunExplicit` и
   `RunExplicitWithDiagnostics`. Backend обязателен, fallback не применяется.
4. Добавить Scalar, Parallel CPU и SIMD in-place Map.
   Реализовано 24 июля 2026 года: добавлены `RunInPlace` и
   `RunInPlaceWithDiagnostics`, возвращающие исходный массив. Auto выбирает
   SIMD, Parallel CPU или Scalar без GPU. Диагностика содержит `IsInPlace`;
   отмена или ошибка не откатывает уже обработанные элементы.
5. Добавить GPU in-place Map для помещающихся данных.
6. Добавить последовательный chunked GPU Map и Zip.
7. Добавить chunked in-place Map и Zip.
8. Добавить chunked Reduction.
9. После Histogram добавить chunked Histogram.
10. Добавить copy-on-write для in-place операций execution graph.
11. Добавить streaming с несколькими buffers только после benchmark-проверки.

Полноценная поддержка произвольных C#-методов на GPU не входит в эти этапы и
требует отдельного source-generation или kernel-registration проекта.
